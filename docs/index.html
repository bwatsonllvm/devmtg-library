Prologue

Foreword

The LLVM Research Library is a static, public research index that unifies talks, papers, blog posts, and people records into one search surface. It is intentionally designed for two reading styles: quick lookup for newcomers and deep retrieval for advanced LLVM contributors.

This page is intentionally structured like a book table of contents: you can read sequentially, jump chapter-by-chapter, or filter the sidebar to jump straight to a specific subsystem.

Mission and Scope

The library indexes four public content families: LLVM Developers' Meeting talks, LLVM-related papers, LLVM blog posts, and derived people records (speakers plus authors). It is not a replacement for canonical event schedules, but a fast research layer over trusted public sources.

Core Objective

Make LLVM knowledge retrievable by topic, person, venue, year, and phrase-level search intent.

Primary User Modes

Global query discovery (work.html?mode=search) and entity rollups (mode=entity).

Audience and Reading Paths

  • First-time contributors: read Prologue, Search Architecture, and Query Syntax.
  • Data pipeline maintainers: start at Talks Dataset, Papers Dataset, and Operations.
  • Frontend maintainers: start at Result Controls, URL State, and Engineering Guide.
  • Release owners: focus on Validation Gates and Release Flow.

Part I

Search Architecture

Search is route-aware and relevance-first. Section pages default to local filtering for browsing, while global free-text intent routes to work.html?mode=search. Shared ranking helpers in js/shared/library-utils.js provide cross-type consistency.

  • Home/About/Detail pages default to Global Search routing.
  • Talks/Papers/Blogs/People section pages default to local scoped behavior.
  • A global-search action is still available from every section search bar.

Autocomplete and Routing

Every .global-search-form is hydrated by shared autocomplete logic from js/shared/global-search.js. Suggestions are grouped by key topics, people, talk titles, and paper/blog titles.

Key Topics Speakers + Authors Talk Titles Paper + Blog Titles Run global search action

Relevance Model

Relevance ranking applies normalization, synonym handling, phrase parsing, fuzzy tolerance, and field-weighted scoring. Broad terms are de-emphasized when narrower terms are present, and low-confidence tails are pruned on broad queries.

  • Weighted fields vary by content type (talks vs papers/blogs vs people).
  • Tie-breaking includes recency and citation-aware boosts where appropriate.
  • Positive and negative constraints are applied during scoring, not only in post filters.

Advanced Query Syntax

Global Search supports explicit operators for authorship, topic matching, venues, year ranges, and exclusion tokens. This syntax works both from manual query input and the Advanced panel controls.

"MLIR-based code generation" author:chris lattner venue:arxiv year:2022-2025 since:2025 topic:mlir -cuda

Result Controls and Views

Each route exposes mode-specific controls while preserving common semantics. Talks uses relevance/newest/oldest/title; papers and blogs add citation-aware sorting; Global Search supports cross-type scope toggles and time windows.

  • Section pages support grid/list density switching.
  • Global Search supports expanded/compact result density.
  • Advanced panel state is explicit (Advanced On/Off) and URL-persistent.

URL State Contract

Search and filter state is URL-backed so shared links are reproducible. Global Search routes preserve query, scope, time window, year bounds, and structured advanced fields.

work.html?mode=search&q=mlir&scope=all&time=since-2025&sort=relevance&view=expanded work.html?mode=entity&kind=speaker&value=Chris%20Lattner
Part II

Talks Dataset

Talk records are synchronized from llvm-www/devmtg into devmtg/events/*.json and normalized to preserve consistent fields: IDs, meeting metadata, speakers, slides/video URLs, categories, and key tags.

  • Event manifest and versioning live in devmtg/events/index.json.
  • Canceled meetings are preserved as metadata but excluded from active meeting counts.

Papers and Blogs Dataset

The canonical papers database merges LLVM publications, OpenAlex discovery, and LLVM blog sources. Final build output is published to papers/index.json with a versioned source file.

  • OpenAlex metadata refresh includes title, abstract, authors, affiliations, citations, and URLs.
  • Blog entries are tagged and remain queryable alongside papers across all search surfaces.
  • The final merge stage is the authoritative source for on-site paper/blog records.

People Index

People records are runtime-derived from talks and papers, then merged by normalized name keys and middle-name variant handling. This keeps the people catalog connected to fresh source content without requiring a separate hand-maintained database.

Update Log

updates/index.json tracks new records and metadata deltas: newly added talks, slides/video additions, new paper entries, and blog additions. Entries are presented newest-first for release visibility.

Scope and Limits

All indexed material is public. The site is a research index, not a replacement for canonical schedules or source platforms. External URLs can change, disappear, or redirect; downstream metadata completeness varies by source.

For canonical meeting schedule details and announcements, use the official LLVM Developers' Meeting archive.

Part III

Automation Workflow

Scheduled sync runs through .github/workflows/library-sync.yml and opens refresh PRs when source data changes. Automation stages include talks sync, OpenAlex refresh, blog sync, canonical merge, PDF backfill, and update-log generation.

Validation Gates

Two validation layers run before deployment: code-quality checks and bundle/data checks. Search behavior regressions are covered by deterministic Node tests.

python3 -m compileall -q scripts ruff check scripts bash -n scripts/*.sh node --test tests scripts/validate-library-bundle.sh

Release and Publish Flow

Validation and deployment are split across PR checks and the pages publish workflow. A data-refresh PR is expected to pass quality gates before merge; publish jobs then serve static assets and datasets.

Part IV

Repository Layout

Route files live as static HTML entries at root and section directories. Shared assets are organized by purpose: css/ for design system, js/ for route behavior, scripts/ for ingestion, and JSON bundles under data folders.

  • index.html, work.html, talks/, papers/, blogs/, people/, about/, docs/, updates/
  • devmtg/events/*.json for talks and meetings
  • papers/*.json for source/derived paper bundles
  • tests/ for search regression checks

Script Catalog

Operational scripts cover synchronization, enrichment, validation, and normalization. Typical entry points include sync-*, build-*, backfill-*, and validate-* families.

Sync + Build

sync-devmtg-from-llvm-www.py, sync-blog-from-llvm-www-blog.py, build-single-papers-db.py

Validation + Quality

validate-code-quality.sh, validate-library-bundle.sh, deterministic search ranking tests

Testing and Quality

Search relevance coverage is treated as deterministic behavior. Tests in tests/search-ranking.test.cjs verify ranking and query-handling invariants so relevance tuning remains intentional and reviewable.

Extension Playbook

  1. Update source ingestion scripts and dataset manifests first.
  2. Regenerate canonical JSON bundles and update index manifests.
  3. Wire new fields through shared rankers in js/shared/library-utils.js.
  4. Add regression coverage in tests/ before tuning UI behavior.
  5. Verify route-level URL state and shareable links.
Appendix

Canonical Routes

  • index.html — Home
  • work.html?mode=search&q=... — Global Search
  • work.html?mode=entity&kind=speaker&value=... — Entity rollup
  • talks/, papers/, blogs/, people/ — section indexes
  • about/, docs/, updates/ — support/reference routes

Glossary

  • Global Search: cross-type relevance mode on work.html?mode=search.
  • All Work: entity aggregation mode on work.html?mode=entity.
  • Scope: content-type focus within search mode (all, talks, papers, blogs, people).
  • Coverage-aware gating: logic that lowers broad-term dominance when narrow terms exist.
  • Canonical merge: final build stage that produces the site’s authoritative papers/blogs bundle.
