66// Define some data structures to describe known plugin "namespaces".
67// The PluginManager is organized into a series of static functions
68// that operate on different types of plugins. For example SystemRuntime
69// and ObjectFile plugins.
70//
71// The namespace name is used a prefix when matching plugin names. For example,
72// if we have an "macosx" plugin in the "system-runtime" namespace then we will
73// match a plugin name pattern against the "system-runtime.macosx" name.
74//
75// The plugin namespace here is used so we can operate on all the plugins
76// of a given type so it is easy to enable or disable them as a group.
77using GetPluginInfo = std::function<std::vector<RegisteredPluginInfo>()>;
78using SetPluginEnabled = std::function<bool(llvm::StringRef, bool)>;
79struct PluginNamespace {
80 llvm::StringRef name;
81 GetPluginInfo get_info;
82 SetPluginEnabled set_enabled;
83};

403 /// \param[in] debugger_init_callback

405 /// instance is initialized.
406 ///
407 /// \param[in] filter_callback

409 /// launches a process for debugging. The callback must
410 /// do the following:

414 /// can be inspected as needed. The ProcessLaunchInfo is

417 /// are needed to support the feature when enabled.
